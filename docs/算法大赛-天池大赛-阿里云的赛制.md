## 3. 一、赛题背景

大语言模型（LLM）正从“能对话”向“能做事”演进。在科研辅助、深度学习、商业调研等高阶知识工作场景中，用户提出的问题往往具有 **多跳推理、信息验证、工具依赖** 等特征——仅靠单轮问答无法满足真实需求。

为此，业界提出了 **Research Agent** 的概念：一个能够自主规划子任务、调用外部工具（如联网搜索、代码执行、知识库检索）、整合多源证据并生成可核查答案的智能体,强调 **真实世界问题解决能力** ，要求模型具备推理、浏览、工具调用与答案归一化等综合技能。

然而，构建一个 **工程可用、稳定部署、效果可复现** 的 Agent 仍面临巨大挑战：

- 如何在不微调模型的前提下，通过Agent工程提升任务完成率？
- 如何将工具调用、中间推理、证据整合等环节模块化、可观测、可调试？
- 如何将整个 Agent 打包为标准 HTTP 服务，供自动化评测系统调用？

本赛题要求参赛者在 **阿里云 PAI-LangStudio**  平台上，基于 Qwen 系列基础模型，构建一个端到端可部署的 Research Agent，并通过 EAS 服务暴露标准 API。目标是考察选手在 **Agent 规划 + 工具集成 + 工程落地** 三位一体的综合能力。

## 4. 二、赛题设计目标

1. 引导参赛者掌握 **PAI-LangStudio 应用开发全流程** 。
2. 在 **禁止模型微调** 的约束下，考察选手如何通过工具策略、提示词优化等方式，最大化基础模型在复杂任务上的表现。
3. 建立可量化、可复现、可审计的评测机制，推动 Agent 架构创新与工程实践标准化。

## 5. 三、赛题具体内容与评分标准

### 5.1. 任务要求

#### 5.1.1. 任务概述

参赛者需在 **PAI-LangStudio**  （工作流模式/代码模式）中开发一个 Research Agent，并将其部署为  **PAI-EAS HTTP 服务** 。该服务需能接收自然语言问题，返回文本形式的答案，且 **仅依赖允许的模型与工具** 完成任务。

赛事分为两阶段：

- 阶段一：提供 100 道公开问题作为验证集，参赛者在此数据上调试 Agent，提交答案，排名前60%的队伍进入下一阶段，此时需要提交 Agent 服务和源码，后续不可更新。
- 阶段二：使用 100 道非公开问题作为测试集，对参赛者的 Agent 服务打分，得出最终排名。阶段二时间窗口内，每个 Agent 只有一次被测试机会，如果测试时Agent 服务异常，视为零分。 **如过程中发现作弊行为，取消成绩并做公示！详情见第五节竞赛公平性与开放性说明。**

#### 5.1.2. 评测数据集

- 每题拥有 **唯一答案** ，题目类型涵盖事实验证、多步推理、单位换算等， **仅需联网搜索即可解答** ；
- 如题目中声明了回答格式，请严格遵循，否则会被判断为错误回答；
- 如题目中没有特殊声明，则答案和问题的语言保持一致。例如中文问题，答案也需为中文；
- 第一阶段使用 **100 道题目：** [下载数据集](https://pai-quickstart-predeploy-hangzhou.oss-cn-hangzhou.aliyuncs.com/tianchi/question.jsonl) ；
- 第二阶段使用 100 道题目作为测试集，赛事结束后公开题目。

#### 5.1.3. 交付物要求（强制）

- **第一阶段**
	- 验证集在自己 Agent 上的结果，文件格式 JSONL，内容形如：  
		其中每行是一个 json，包含 id 和 answer 两个字段，表示评测数据集中指定 id 的问题的答案。
	```
	{"id": 0, "answer": "answer_1"}
	{"id": 1, "answer": "answer_2"}
	{"id": 2, "answer": "answer_3"}
	```
- **第二阶段**
	- **EAS 服务** ：部署成功的 HTTP Endpoint 与调用 Token；  
		LangStudio 应用流部署为 EAS 服务后，通常以 **公网Endpoint + Token**  调用，并通过  `Authorization: Bearer <token>` 传递鉴权信息。([阿里云帮助中心](https://help.aliyun.com/zh/pai/user-guide/application-flow-deployment))
	- **LangStudio 项目包** ：LangStudio 支持导出应用流（应用流 ZIP），可作为交付物之一。([阿里云帮助中心](https://help.aliyun.com/zh/pai/user-guide/application-flow-development/))，项目包中请包含说明文档（README），提供基于前述项目包的全部复现步骤，包括设计架构、依赖说明、调用示例等。
	- **参赛者承诺：请保证EAS 服务是通过LangStudio 项目包部署而来，赛事主办方会做检查，如果发现 EAS 服务中有不属于项目包的逻辑，取消成绩！**

### 5.2. 功能与性能测试机制

#### 5.2.1. 评测流程

**第一阶段**

1. 评测脚本持有 100 道验证集问题及其参考答案；
2. 将参赛者提交的答案和参考答案进行归一化匹配
3. 统计成功回答的题目数，计算 **正确率** 。

**（截止时间2026年3月9日）**

**第二阶段**

1. 评测脚本持有 100 道内部测试集；
2. 时间窗口 3 月 9 号到 3 月 13 号；
3. 对每个问题，向参赛者 EAS 服务发送 POST 请求（带 Bearer Token，流式请求）；
	- **输入示例** ：单个自然语言问题，通过 HTTP POST 请求传入，字段名为 `question` 。
```
curl -X POST \
     -H "Authorization: Bearer <your_token>" \
     -H "Content-Type: application/json" \
     -H "Accept: text/event-stream" \
     -d '{"question": "Where is the capital of France?"}' \
     "<your_endpoint>"
```
- **输出示例** ：要求返回响应的Respone内容是JSON字典，问题的答案对应的Key是"answer"。
```json
{
  "answer": "巴黎"
}
```
1. 统计成功回答的题目数，计算 **正确率** 。
2. 特殊注意：
- 答案必须为 **纯文本** （如 `"140"` ）；
- 超时（>30min）、非 2xx 响应、JSON 解析失败等均视为 **错误回答** ；
- 验收时间窗口的开始/结束由主办方定义。

**主办方将选择第一阶段中（截止至2026年3月9日）排名前60%的队伍，根据选手提交的 EAS 服务信息、LangStudio 项目包、工程 README 文档，进行第二阶段正确率再次验证，以主办方验证后的正确率为最终结果，并于2026年3月13日公布结果。如未按要求提交，则无法进行第二阶段正确率再次验证，将不参与最终排名。**

#### 5.2.2. 答案判定规则（归一化后比对）

为减少格式差异导致的误判，采用以下预处理：

- 转小写；
- 去除首尾空格；
- 数值题均为整数；
- 如果答案包含多实体，请遵循英文语法，逗号或分号后接空格，具体符号以题目要求为准。

### 5.3. 提交方式与代码管理

第一阶段截止前提交： **（截止时间2026年3月9日）**

1. 评测数据集在自己 Agent 上的结果，文件格式 JSONL

第二阶段截止前提交： **（截止时间2026年3月9日）**

**请参赛队伍务必提交以下信息，如未提交，无法进行第二阶段正确率再次验证，将不参与最终排名。**

1. **EAS 服务信息** （Endpoint + Token）；
2. **LangStudio 项目包（ **LangStudio 导出的应用流ZIP文件，提交前请在LangStudio页面：新建应用->从 OSS 创建-> 选择该 ZIP 文件，创建新的应用流，检查该 ZIP 包创建的应用流是否符合预期** ）** ；
3. **工程 README 文档** （托管于项目ZIP 包内）。

### 5.4. 赛题评分标准

**评分指标：正确率**

- `Score = Correct / N`
	- `N` ：评测题目总数（约 100）
	- `Correct` ：在评测窗口内判定为正确的题目数

其中 `Correct` 为归一化后与参考答案完全一致的题目数。

**附加说明** ：

- 不设“部分得分”或“模糊匹配加分”；
- 若发现硬编码、模型替换、第三方 Agent API 调用等违规行为，该题得分为 0，严重者取消资格。

## 7. 五、竞赛公平性与开放性说明

为确保公平、可审计、可复现，设定以下 **强制约束** ：

### 7.1. （1）模型与数据红线

- **禁止任何形式的模型微调** （包括 LoRA、Adapter、自训练权重等）。
- **仅允许使用以下模型服务** ：
	- 通过 [PAI Model Gallery](https://help.aliyun.com/zh/pai/user-guide/model-gallery/) 部署的 Qwen 基础模型；
	- 通过 **阿里云百炼** 提供的 Qwen 系列模型服务。
- 禁止在提示词、Python 节点、配置文件中 **硬编码评测题答案** （包括哈希映射、关键词触发返回固定答案等）；
- 允许使用阿里云 IQS、SerpAPI、Bing 等任何不包含 Agent 能力的搜索引擎、允许使用 MCP。
- 禁止调用任何第三方 Agent 服务，所有 Agent 逻辑必须在 LangStudio 项目包中实现。

### 7.2. （2）项目审计机制

- 评委会将使用自动化脚本扫描项目配置与代码，检测硬编码、外部调用等违规行为；
- 请保证阶段二中，EAS 服务是通过LangStudio 项目包部署而来，赛事主办方会做检查，如果发现 EAS 服务中有不属于项目包的逻辑，取消成绩；
- 发现其他作弊行为， **立即取消成绩并公示** 。

### 7.3. （3）引用与开源规范

- ✅ 允许参考开源 Agent 设计思想（如 ReAct、Plan-and-Execute）；
- ⚠️ 若参考特定项目，须在 README 中 **明确注明来源** 。
